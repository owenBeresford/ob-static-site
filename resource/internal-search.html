I am mostly an App developer and REST API creator. I have fairly large experience in making search tools and information retrieval, which I pull out into one article as this has come up in discussions. The list is time sorted at present. The scale factors at the start of the items show that I have most experience in micro-startups. I think good use of data is important, and add features to support use. Obviously scale numbers may have changed since I did that work.

Worklog for search:

  • My first degree dissertation was on information retrieval using four value logic where I extended a PhD student's work, and the second degree dissertation would use many information retrieval features inside a different UI for a virtual world simulation.
  • I have improved search and information retrieval features at many companies I have worked for. Note across time, capacity for larger numbers is easier as cloud services, and as PC hardware is generally faster. Please review the year in each case ::
  • (<400,000 requests during the hot last working day of the month, then untouched for the next 28 days) I built information retrieval for billing systems that was mostly only used for a few hours per month, but reported on the entire enterprise's activities. From this ISP, the data displayed customers' internet usage, with traffic breakdown. Much of the retrieval was pre-built into summaries across the month, and the business processed over 1/3 million monthly receipts from end users. The starting raw data was ~1,000,000 rows per hour, this is the limit that those 2007 edition servers could handle (32bit CPU) and I implemented sharding on the time-stamp. The output was a short Excel file with many numbers translated into “human values” (e.g. “streaming video”, rather than a TCP port and a UDP port). System offered an aggregation on “long tails” feature (think Bell curves), to merge all the small values. MurphX.
  • ( <1,000 request per day) I used standard RDB instances and optimised the schema mostly adding indexes. Small features at Matatiro solutions, Hoptroff, and assorted contacts.
  • My boring website has integration with a search engine, to help users find content on it.
  • (<100,000 requests per day, noting bursty usage) I used standard RDB instances, optimised the schema to the point of adding denormalised caches. The traffic monitoring showed that most users were between 17:00 and 20:00. The data was displayed as heatmaps, frequency across time or location. I ideated with slider inputs for numerical range data before I saw sliders in many mobile apps in 2012. MotorTRAK
  • (HOPEFULLY <1000 requests per day) Search feature to support incoming calls on standard RDB with data volume in the order of 100,000 accounts. RDB wasn't stressed and was used for other business activities. The search identified accounts from ~10 possible identifiers. The focus on speed of results, and speed of comprehension was essential (as live phone call). I added a few extra “rich” indexes to allow the fairly large number of queries per user input to work promptly. These indexes included email domains-only, and a normalisation process for phone numbers. I had plans for a GIS lookup but this proved unnecessary. Search needed consideration for “safety” in terms of GDPR 2014, FairFX
  • (<1000 requests per day, but that was before users) Prominent search feature at a startup, displayed complicated search results across a possible input of 10 variables, against a mostly static dataset of 500,000 documents. One search limit was the current location, either address or long/lat from a phone. I organised the API to return some results quickly then retrieve more data on a bigger radius if the location wasn't moving. Many other filters were effectively array matching subqueries per document, where the underlying data included a list of values (eg multiple cuisine types per document). My time included improving the data population in many places. This service was supplying useful data whilst browsing on the tube, as real-world testing. My technology was better than the achievements of the sales team. 2015, Venue10.
  • (<100 user sessions per day, each user session may be a 100 graphs to a very large number for wall displays). “Predictable search” for data aggregation to draw real-time graphs. Useful graphs are an hour or a day in duration, but my platform could do up to monthly graphs. Several clients had wall displays permanently running. Sub-sampling so all data displayed could be seen, just like AirBnB. Average graph load time was ~1s unless browser was in a different continent to the DB server. MongoDB cluster, with each client being held as isolated data. Each continent's accounts was held on a cluster nearer to them. Hoptroff.
  • For the same user-journeys on an RDB, the multiple tabs saturated the DB and there was many DB locks lasting longer than desirable. To avoid the locks, I added an extra de-normalised cache table. Data existed in this cache for an hour, then removed. This cache, and some better queue management on the client side, reduced a worst case cold-start page-load from 30s down to 3s from the user perspective. The queue management meant that data users could see arrived first. Hoptroff.
  • Prototype built with Redis and Kafka, in case we had an international scale client, was running smoothly with 100,000,000+ new documents per month, but needed to change the schema to suit Redis.
  • (<100 requests per day) Monthly compliance report, that aggregated an entire month's activities. Mostly only used 1 day per month, and outside of working hours, as it was heavy reporting. The report ran about 100 aggregate calls, that each would impede normal business (data logging) operation. Aggregated data source was about ~60,000,000 new documents per month. Elastic Search cluster, with geography-based horizontal-scaling again. Known as product3, at Hoptroff.
  • Repeat last feature, but in a different product, and MongoDB cluster running at ~20,000,000 new documents per month. Known as product1, Hoptroff.
  • (I saw <1000 requests per day, before the service was published). Most frequent user interaction, a search feature, hosted on RDS. I rewrote an earlier version of this as it locked the tables. ~6 input variables. To scale better, it needs to be data architecture, but that is in the future, Swyvl.
