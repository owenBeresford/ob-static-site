    A foldable menu

    Pikka boo! Focus point sorry for abuse of ARIA attrib on this link



    Technical Notes

    Inventing your own interaction is rarely a good idea as even if you are genius, your users do not excitedly learn your UI/ tools.
    BUT
    I really need less long lists with keyboard access. This Component / block idea means the UI is much faster to use.

    Desired behaviours

    • I have implemented the recommended spec on w3c, I don't think this helps mobile/ haptic users (tab key? arrow keys?).
    • DEFAULT for mobile can use touch to open menu, but cycle selection modalities don't make sense
    • can cycle components / blocks by tab key
    • can cycle links or other single values by UP / DOWN arrow keys
    • can trigger the links with SPACE or ENTER (or touch on mobile)
    • ESCAPE should close a menu (added item, as it common in other contexts)
    • no item is focussed by default on page load
    • shift-tab works as win32

    Functional spec

    • Work out if faster to auto-append “addKeyboard” CSS class to components, or leave it as hand mapping, so less scripting needs to be created.
    • Create interstitial with a guide text, launch it ONCE when tab or arrow keys are used outside of text entry elements
      • It MUST have a visible close to remove it
      • Think about cookie for the above interstitial, so its only shown once per month by default
    • Build process to apply to each component and as real/ tidy code
    • Make above interactions nice
    • Iterate UI feedback enough, think about colour abit more
    • Something to view help/ alt texts via keyboard access

    Local site components to affect if this adds value

    • SM share-bar, mobile and bigscreen
    • article menu
    • article segments menu
    • links in footer component
    • links in adjacent content component

    • Permanent note on anything to do with UX, interaction design or usability: provide a revert OR cancel interaction, so mistakes are reversible. Intro article 9.

    NOTE: using enough Aria attribs increases intent and accessibility for a wide range of users. Also using too many Aria attribs adds no value and may confuse the poor screen-reader. Whats the correct magic number of attribs? Off the cuff: it depends, I think it varies per user.

    I see threefour possible solutions

    • A keyboard menu structure, context box thing, like actual apps or console game UI. [not very good on mobile]
    • Direct key bindings, like in win32 Apps, save is often <cntl-s> [not on mobile as chords]
    • A arrow key cursor, so re-inventing an awkward mouse [not on mobile, AWKWARD]
    • A minimal scripting language, a la vim or emacs. [awkward on mobile]

    For direct keybindings, this is not really guessable/ usable for websites 15. For an App, heavy users might be able to gain, but how to make Chord sequences? Do you, reader, use <cntl-B> in msft word to bold something, or a labourious menu crawl as its now Tuesday and MSFT have moved all the UI round again. The tech is 16 17 18 19. Everything I read on this option mentions manual testing, as JS doesn't allow real keyboard events to be made (#leSigh). Accessibility aXe (testing for keyboard input) 20. It is necessary to have a visualisation process for your keyboard assignments. Awareness of this API 21 when it becomes more widely supported is advised.

    Writing a language is a oversize scope solution. Even less guessable that the keybindings. Here, state your goals to the app, and let the software work it out, eg with a defined list of terms, so “share this to xitter”. However see initial comment that no-one values a single system that much, unless they use it everyday.

    This INTERSTITIAL can be closed by the X on the right. It will only appear rarely (and if cookies are wiped)
    You seem to be using keyboard access as an input modality.

    • Using the keyboard on text input elements, works in normal fashion.
    • Using the TAB key will select a component, and adds a border to it.
    • The UP / DOWN arrows will cycle selectable items in that component.
    • SPACE or ENTER will trigger the selected item
    • This sites uses many DETAILS/ SUMMARY HTML elements, these should open/ close in a helpful fashion
    • The technical HTML structures should be hidden from you in the component choices, as that is not your problem.
