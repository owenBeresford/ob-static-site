Updated source in new TS project

I read advanced tech blogs, written by people similar to me. I gain perspective on this. I note that time to read the article is quite useful, so I have mocked up a JS tool to supply this. This is the first iteration, I may add more metrics to the output in future. My inspiration is 1, 2 and Medium 3, although I am sure there are more examples. UPDATE: According to 4 reading time is very similar regardless of device screen-size. I would take 5 as a better reference, as it lists more details. However the last proposes 238 not 275.
For authorship, I would advise using a article content-assistance tool, rather than my simple notes. Some examples are Hemingway App, Pro writing aid, grammarly. These are a more general solution to not having spent fifteen years grinding out text everyday. If you are using Wordpress, there is a plugin inside that framework.

Model

The average reading speed is quoted as being between 250 and 300 words per minute 6 7 8 9 10. I also find a separate ranking scheme, IELTS 11 12. My casual survey did not find any breakdown by subject, Fogg value, nationality, primary or secondary language status, or educational attainment of the reader. For a more comprehensive tool this analysis is required. In my personal case the biggest routine variance is my fatigue and the language of the text. I suspect but would be very careful in stating, that some languages are better designed for volume data transfer than others.

So a simple reading duration would be expressed as :
var duration= (($(root_element_for_text).textContent .match(/\b[^ \t\n]+\b/g).filter( (a, b)=>{ return !a.match(/^[0-9]{1,2}$/); }) .length)/275) * 60;

I am asserting that people will read most of the text, and not much of the headers, menus, legal disclaimers etc. I am assuming that most webpages have a single root element (be it <article> or a <div>), and that avoids the other visible words.

I will copy Medium's notes 13, and add time for images. These will also be counted from the root element.
//... cont from previous code block + ($(root_element_for_text) .find('img, picture, object') .length) * 12 // code blocks will read slower, so are counted again ... // that's 3times in total + (($(root_element_for_text). find('code').textContent) .match(/\b[^ \t\n]+\b/g) .length) * 2;

This model currently doesn't measure page loading time. That is very dependant on situation. Hopefully this never adds more than 2s to page viewing time, even on a old phone. I have reasonable scores in PageSpeed, so I hope I am clean on this metric.
UPDATE 2024: the code samples based on legacy jQuery are samples, the current code isn't using jQuery, but many people can read this notation.
I have rebuilt the computation algorithm to better model code-samples as I use more of them that to start with. My rule for images is: Images that are content must be visible in the DOM tree. Images that are decor, shouldn't be in the DOM as elements. This means print view will show content images, screen readers see the content images, and awkward phones in power-saving mode show content images. It would be cool if advertisers undertook this classification, but I'm not holding my breath.

Implementation for this simple JS tool

I am creating a jQuery plugin, hosted in github along with my other source code. This plugin uses callbacks to make HTML, for better branding, same as my other plugins (or its not usable by anyone else). Very quick implementation, so this text is short.The legacy code is deprecated. I am using normal JS on this site. This article is left up, as the notes on behaviour haven't changed

DEPRECATED API:

  • readingDuration(opts);

Arguments :
The opts hash may take the following:

  • debug ~ default: 0 ~ how much information to dump to the console
  • linkTo ~ default: null ~ if you want a full page rather than popup, set this to your page,
  • dataLocation ~ default: '.blocker' (to suit me) ~ what element to take as the source of the raw data
  • target ~ default: '#shareGroup' (to suit me) ~ where to put the generated text
  • timeFormat ~ default 'm' ~ US people like putting Dates in a weird order, so that may be set by the dev
  • wordPerMin ~ default: 275 ~ the number of words that it is expected that a person can read per minute. Change if you know the audience is abnormal (i.e. expected age is 5 or all PhDs)
  • refresh ~ default: 0 ~ whether to delete the contents of the target first? Useful if this is a repeat call, as you changed the page content
  • callbacks ~ defaults :

~ shortDisplay
~ longDisplay
