After sept 2025, the newest official release is v18 1 2, prod servers are often 2 versions back, unless a critical feature is added more recently. Installing the docs package is recommended, although maybe to a dev laptop rather than the DB server. According to 3, several performance improvements have been released in v18.

  • From psql, there are SQL keywords, schema items, functions and shell features. Shell features tend to start `\`. Programming languages will support SQL, and possibly the functions. A psql manual is 4 5
  • Postgres has static typing on data 6, but good conversion capacity. The types are extendable if this adds value. To quote
Data types are divided into several basic type categories, including boolean, numeric, string, bitstring, datetime, timespan, geometric, network, and user-defined.
  • There is a `::money` type, with controlled precision 7 (fairly common in DB but avoid ‚Äúcent slicing‚Äù frauds).
  • To perform casting / type assertions, use `CAST ( expression AS type )` OR `expression::type` in the query 8.
  • To get best value from larger text fields, you probably want ‚Äúfull text search‚Äù 9 to easily support partial matches on large fields. Postgres supports ‚Äúfull text‚Äù (this is classical by now) and tsVector matching 10. Dictionaries for Stemming, stop-words, etc should be applied for good quality outcomes 11. Vectors have additional syntax 12.
  • Postgres runs queries in the most CPU efficient method according to the data. It doesn't set any sequencing for sub-expressions (unlike Clang family) 13. Sequencing can be forced with careful use of a SWITCH statement.
  • Syntax features not found in basic webpages describing SQL92
    • In line with other SQL, strings should be delimited with appostrophy / single quotes 14, if a string contains a single quote, double it. Several strings seperated by line breaks will be joined unless there are spaces or similar in addition. As C++ lang, Python or Perl, there are some marked up string notations ~ strings starting with an 'e' have escapes expanded to their real value. Unicode strings are written with 'U&', Bitstrings (of only 0 or 1) start 'B', and un-adorned strings can use '$' in place of '.
    • Numbers can be specified: 0b100101, 0B10011001, 0o273, 0O755, 0x42f, 0XFFFF, 42, 3.5, 4., .001, 5e2, 1.925e-3 without quotes. Underscores can be used for readability same as JS or Python.
    • UUID are a local feature `gen_random_uuid` 15. There is `pg_uuidv7` plugin 16 or `uuidv7` (native in v18), which has better characteristics for databases as it's closer to a time-stamp. See 17 18 for notes on versions, UUID have a public spec.
    • A recent SQL feature 19 is DEFERRABLE on constraints, which allows invisible states when a single SQL statement is executed to break constraints ~ the data is correct before and after, just not during. Available in Postgres 20.
    • On a reporting-only server, it's possible to make queries parallelisable, ie to use multiple vCPUs 21. Read the ‚Äúwhen‚Äù sub-page before being excited. Suspect step0 is to check disk-caches everywhere carefully. I annotate reporting-only, as any type of OLAP or OLTP implies many DB clients, so vCPUs shared between these clients.
    • There are GENERATED columns 22, which are like a view for that column. They may be stored or virtual. Virtual seem as a Lamda function in a programming language. Generated columns are omitted from some data transactions like partitions, or referencing other generated values. Use-case: `name varchar generated always initcap(first_name) + " " + initcap(family_name) virtual` or transforming dates.
    • Operator precedence chart (its more complex than BODMAS) 23.
    • SCHEMA are like a directory to put SQL TABLES in, to make name-spacing more readable 24. For example, a product has SCHEMA 'active', and a SCHEMA 'archive', data is migrated into archive when it is too old for day to day use. There is a SCHEMA called 'public' where items will appear by default.
    • Postgres supports `COMPOSITE` types, which handle like Clang structs, or JSON objects 25. To insert a composite struct, use ROW
      use COMPOSITEs -- https://www.postgresql.org/docs/16//rowtypes.html CREATE TYPE inventory_item AS ( name text, supplier_id integer, price numeric ); INSERT INTO mytab (complex_col) VALUES((1.1,2.2)); UPDATE mytab SET complex_col = ROW(1.1,2.2) WHERE ...;
      . Also `RANGE` types, and non-stupid range block handling 26.
