The answer to “Why MongoDB?" is “because of documentation”.

The wider question being: “why NoSQL?", but MongoDB is a good concrete example. RDB are an established solution to managing and reporting volumes of data on slow hardware, needing access restrictions. This is an introduction to the concept of nosql db and for a couple of later articles (see links at end).

The first version of SQL was built in the 1970s, targeting 1970s machines. The operational platform for information systems is about 10^3 times faster than in the 1970s [ 1 ]. Firstly, we have new faster hardware so why do we need architecture like the SQL system from 50’s – 70’s? In fact SQL works better with faster hardware. A byte of storage cost around 1p back then. Today the cost of storage has dropped by a factor of 3: it just means that today you optimise different dimensions. Strict control on data structures and types is less important, when one quotes “computer operations per second” in 10^9 and human operations per second ~ unchanged since the 1970s ~ at about 10^-2 (e.g. 1 thing per 100seconds ). Structuring concepts in mutable 'documents' rather than normalised 'entities' achieves a functional solution faster. Computing now is different to what it was then. If your data looks like a document i.e a big blob of data that you can access via a single key; then the advantage of mongoDB wins: it is more efficient for documents. If data looks like excel cells, then SQL is more efficient.
Systems like Mongo are important, as they represent today's data and operating conditions. The second reason for NoSQL is map-reduce operations. That is parallelised crunching operations on >10GB datasets to show trends. Via NoSQL this is much faster to build than RDB; and as RDB aren't generally parallelised nor faster performing.

The amount of data we record is at least 10^3 larger than the 1970s, and we expect to be able to access any/all of it quickly. MongoDB as a data storage was introduced in 2010 1. It was designed as document storage system and for collections of documents. Short for ‘humongous’, it is an open source product and the architecture is different for data storage; going from rigid tables to documents. This closely matches the concepts I have been working with for the last decade, this is why I write about it.
Taking 2 as an example; your entire infrastructure is supposed to be ported to NoSQL. My earlier reading held that NoSQL was very good at certain things, but as RDB were established with a large tool set, they where more efficient.
