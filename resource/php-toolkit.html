PHP the language is documented as aiming to seize market share first, to ensure the language had traction and ensure involved people to continue.

Language target

PHP is a widely used net language, but less dominant than it was 1 2 ~ the second reference is by Zend inc who sell PHP facilities. Those notes do not mention Ghost or similar generation JS blog tools that supersede wordpress. Journalists have been saying its dead for about a decade, but there is much established work running on it. In Feb 2026, the current edition of PHP is 8.5 3. This is a very broad features language, although they have been improving the language architecture for well over a decade. The project docs are quite good 4. Using the newest PHP version is always recommended. By whatever means needed use >PHP7 as it got an AST then and all the standard tools work much better. There are several ways to execute PHP, but it is less linked to clouds than other languages. PHP has language extensions for many tools or services (eg the well-known MySQL).
PHP the language has more complex licensing 5 6 7 than other languages, which I think is an awkward situation. PHP can be compiled from its source, and doing so is recommended for awkward/ non-standard setups.

Engineering details

Other toolkit articles do not have this section. I think PHP needs this list of caveats.

  • PHP is only used as a server side language, and normally used with HTTP variants.
  • It can be run as a persistent interpreter ~ not as a single PHP script ~ with FPM. Or as a persistent interpreter and a single script, with or without POSIX assistance. Historically, PHP was run as a sub-process of Apache, or as a self ending script process like bash scripts are.
    • The first option mentioned is recommended as it reduces time spent parsing code. Typically a single process can be used for 10,000+ HTTP requests, and only parsing the code once saves quite a lot across the duration.
  • Before starting a project in PHP please read the 'fractal of bad design' 8 9 10 11. It is stated that "a bad worker blame their tools", BUT a worker should chose their tools, and avoid the bad ones.
  • PHP has a share-nothing interpreter arch by default. Due to this, it is common to have more than one version of PHP on servers.
    • My local machine at random has php 7.4, 8.4, 8.2 and 8.3 on it. I use some libraries written for PHP 5 that cannot take newest PHP, and I don't have random time to update legacies (it's offline code).
  • To-date PHP does not have real hot loading (unlike Java or C++). Some frameworks may offer it by soft naming after a compile step (what JS does).
    • A second interpreter ~ possibly in a different language ~ is often used as a monitor and can supply the results of hot-loading. State is often stored in a DB between HTTP requests, so if it a different process, nothing was lost.
  • Random Clang modules can be inlined with `dl` 12. Use with caution if essential.
  • Memory handling in PHP is complex and abstracted. PHP uses a COW scheme [XXX], and reference counting 13. Note “easy” loops make a duplicate of the array, so use with caution on large arrays. Memory can be forced to be recycled via 14
  • PHP Arrays are complex creatures and have no relation to Clang arrays ~ think `dict()` in Python, not `list()`. Use of better types is recommended 15 16 17 if possible. Blackfire has a series on optimising PHP including arrays 18.
  • Old PHP had very poor async capacity or poor and unsafe thread capacity, so multi whole process arch was common. Newer PHP can have async docs 19, and can have threads 20, cautiously. DB drivers are not aware of threads, and will need careful management.
  • There are some language features that I think should be seen as dangerous, and marked as errors via a lint tool.
  • The biggest gotcha is that PHP can ignore errors and keep rolling ~ with a malformed app state.
  • There was a hack called “super globals”, and these should be avoided with extreme prejudice.
  • Any known errors can be made silent with an '@' at the start of the line 21. Obviously not recommended.
  • Types on primitive data are flakey, and there is auto conversion ~ which sometimes isn't correct (here primitives includes strings), although this is handled better than earlier versions after PHP7.
    • As a following sentence never use loose-type comparisons, as chaos will ensure with realistic data eg ` ('0e3' == '0e14') ` is true.
    • Also `TRUE` or `FALSE` might show up in some places, and are two ints, not ranges/ bit patterns, singular values. Therefore a thing that evaluates to `TRUE` isn't `true`.
    • Language functions that are shallow wrappers of Clang calls might return `-1` flag value for a error. This shouldn't be naïvely used with a boolean test, as `-1` is `true`.
    • Junior devs are always tripped up by array indexes starting from 0. I try to reduce this by calling them 'array offsets'.
  • String handling is either single byte (ASCII) or multi byte (unicode). These are completely different API. Either avoid low level string manipulation, or use MB functions generally. Perl and Python do this aspect much better.
  • A positive feature of PHP is it can run untrusted scripts if “safe mode” is configured 22 correctly. This must be done from the config file as a security hobble.
  • One of the low level abstraction leakages is “tick handling” 23 24. Only use if necessary, test very carefully, as it will make your production systems fragile.
  • This article format doesn't cover greater scope, I was hoping to learn if anything had moved in the last several years since I used PHP.
  • Long lifespan PHP is generally written in a strict layout, just like Python or TS. However, there are many standards to choose from, index 25. PSR 7, I think the newest code spec 26.

Realistic optimisations for PHP

  • setup opcache correctly
  • rollover to FPM (if absent)
  • check settings for FPM
  • Manage process restarting (FPM or overwise)
  • by some process reduce disk reads to read PHP source. ESSENTIAL for non-FPM proceses
    • can be PHAR packaging, LEGACY zend byte code blobs, or another similar product 7 There have been many PHP accelerators, use a current and active one that supports the current edition.
  • use RAM better, see splfixedarray above
