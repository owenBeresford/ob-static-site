If the text you are writing resembles a blog, custom fonts are nice but not essential. As a rule of thumb, I am sure >50% of users will not notice what font you use. This effect is amplified if the reader is using a small screen like a phone. If a person with an odd setup sees your text with their system default-font, they have still completed their user journey successfully. The common practice of pretending that time doesn't exist whilst a font is computed in a google cache doesn't make sense to me (blocking with no content for about 3s). It does make me leave the site, or at least background the tab. If a user has really old or odd technological choices ~ IMO ~ it's better to use system fonts, their tech choice has said they do not want bells or whistles while they browse.

However there are other uses of fonts, such as icon fonts where it is worth having a very good level of support; as technical failure would break user journeys. With current technology, I sometimes use UTF-8 chars rather than a logo font, as this is less bytes. The logo fonts are awesome as a means to send company logos efficiently. As ever, ‚Äúwork out who your users are and support the right technology for them‚Äù is more important than these market percentage sizes.

It is valuable to test any international site from outside your locale. Leaving your expected alphabet will mean you have a better visibility from another's perspective, and definitely show you sweeping assumptions of your tools. This is now the third article I have on this site about fonts, I think text is more widely used and more important than your favourite video clip. As a design first opinion, please browse 1 2, which amongst other details lists highly successful sites that are mostly text.

Web fonts

By 2020, there are few remaining successful webfont formats; and the need to support multiple choices drops again by 2024.

  • .woff2 (Web Open Font Format) 3 4 only not supported by really old browsers. It is supported by about 98% of ‚Äúgeneral internet‚Äù users. If your audience is ‚Äútech people‚Äù, I'm sure this climbs to 100%. This newer version of WOFF has better compression 5, so is faster to access.
  • .woff (Web Open Font Format) should be used for fonts needed for essential user journeys, like symbol fonts. Use this option to leverage a few more users.

For detailed work, there are two taxonomies for fonts. Traditional ‚Äústandard‚Äù fonts, and ‚Äúvariable‚Äù fonts. For high attention areas of text, it may be valuable to kern/ variable space/ typeset each letter individually, but for general body text, I wouldn't. For 2016 font tech and CSS, please browse 1. Most fonts are not encoded as this is, and the variable fonts seem to be larger files (so slower to access in Apps). Variation version need to be loaded as 'format(‚Äúwoff2-variations‚Äù)', and similar for other formats 2. The theory of variable fonts is that complex documents may need several weights of the letters in the font, and bundling these in the same file saves total disk / bandwidth used.
The WOFF/WOFF2 formats are an umbrella format (again), so different vendors keep their older work, but join in with the new spec. It is possible to have a OTF, TTF or VF WOFF font. The variable fonts (VF ones?) are most relevant to this section, and should be controlled by CSS font-variation-settings. OR There is a different system ~ focusing on numbers ~ font-variant-numeric which seems to need OTF fonts/ encoding.
It is possible to have fonts for specific purposes (e.g. only Japanese letters, and normal numbers), but I think the fonts I used for this article have very large numbers of glyths, to be able to show the new font encoding style reliably. Font optimisation notes 3 which state, good use of VF allows less bandwidth on fonts in total, as having multiple styles in the same file has less duplication. For common styles/ frameworks, using unicode-range 4 5 allows the browser to ignore fonts that do not apply to the current page better. Unicode-ranges can be more easily built by referencing 6 this big table. In terms of logos, a font can be loaded for a few specific letters with unicode-range, but the font would need to be compiled for this purpose to make this bandwidth efficient. For convenient access to font settings, 7 (use advanced settings) saves time, or use fontforge 8, installed locally, for detailed editing. If your starting point is wordpress, try 9. As background reading 10 describes technical options in detail, scroll to ‚ÄúCSS font matching algorithm‚Äù for browser made fonts. The wiki page says to check that fonts are being served with the correct MIME type font/woff / font/woff2. MIME type for WOFF 1.0 was application/font-woff 11.

I haven't built much to run in a webrowser that should be emitted via a printer. If I did, I would use the PDF format. PDF has different font approaches, and is not covered in this article. Please always use the newest available version of PDF format 1, some tools use decade old PDF writers.

So, fonts?

For nearly everything, I use locally hosted fonts, as this is ‚Äúbetter security‚Äù during audits. If you are using CDN fonts, please try the optimisation tips listed on 2 3. My App servers are physically fairly near the clients, a SaaS with a .com domain may or may not be. For a high use services, I would put fonts on a different subdomain, just like pictures. ‚ö† WARNING ‚ö†, readability

There are text focussed CSS declarations that affect the way that the letters look. Editing fonts so they are readable and usable in many different situations is a larger and more difficult skill than adding some CSS. A font can be made italic by setting a CSS declaration 6, but a font won't be readable and italic without more work in a font editor (or using another person's work with the editor, but that does the same thing).
The previous link showing synthesised fonts shows them to be less pronounced than the actual designed ones. In terms of reading accessibility, a human designed font is definitely better. But in terms of reducing bandwidth (so reducing people waiting) the browser made fonts are much better.
In terms of flicker when fonts are applied, this can minimised by smoothing the differences in the default font, and the preferred one. This can be done by setting size-adjust, ascent-override, descent-override, and line-gap-override 7. This is a supportive feature only, as these CSS declarations are less often supported on Safari or MSIE 8. As far as I can tell, these values would need to be computed manually.
