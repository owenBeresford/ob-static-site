I am writing this to show recent experience with a subject that I did cover in my degrees, but those where a very long time ago. This role required project management, SEO and QA concurrently to work on the UI. Please see lead-engineer for the business context. If I had been involved for the first edition, I would have pushed to have these changes in the first edition. Many of these changes are obvious requirements (e.g. personalisation on messaging and magic links).

High level process

  • Define all changes as user journeys, to pull the consumer into the picture.
  • Define technical success metrics that are relevant to the user (e.g. display search results, but do so inside 2s). I was doing much more testing than previous developers.
  • Build several Persona. Ban the word ‚Äúuser‚Äù see why.
  • Think about the user journey, not just the website. Some interactions aren't on the website, but are still marketing presence for our clients.
  • Walk through the platform and fix all the ‚Äúdrop down a big hole‚Äù interaction end points. The consumer still exists after the transaction, and may wish to continue the relationship with the client if they have a good experience.
  • Reduce consumers knowledge requirements, down to what most shops require.
  • Got some new users to perform actions on the platform, I took notes about where it wasn't intuitive.

Requirements and UX observations

  • With the marketing director, make ‚Äúcreate account‚Äù a less obstructive action, so that it doesn't have to be done first. This moved the site closer to normal e-commerce.
  • Make the UI less interrupting ~ e.g. message prompts ~ so casual consumers aren't nagged all the time.
  • With the CEO design a landing page, so casual consumers can decide inside a few seconds if the platform is useful to them. See ‚Äúinformation scent‚Äù 1 2
  • Make outbound communications personalised and relevant, so users can use the email/SMS for any actions. I wrote all of this bit.
  • Emphasing ‚Äúcasual user‚Äù again, make ‚Äúmagic links‚Äù for emails, embedding user session, so the email is useful, and doesn't require re-authentication (my code again).
  • Push for RWD and mobile on main site. Over time organise rewriting most CSS in the whole platform. Before I joined there was an iphone app, but it wasn't nice UI, it was too slow and it covered few important user-stories.
  • Completely redesign the wishlist interactions (in 2014, you need 5 touches to add a venue to your wishlist; after me, 1 touch).
  • Rebuild outbound links including social media, as many people will be going with friends and need to talk about it to them. Update list of social media to reflect current platforms.
  • Completely rebuild venue filters to be dynamic, reflecting current data. This makes them much more useful for filtering. Due to staff holidays, I also wrote all of this JS.
  • Completely redesign the ‚Äúvenue edit‚Äù features, and create a ‚Äúcreate venue‚Äù feature. Create a user flow from the users perspective, rather than a set of disjoint pages. Unless cookies are wiped, most clients and consumers will only need to login once.
  • In a change involving everyone, rewrite image gallery UI. New UI is much easier to use, with less steps and is more user-centric.
  • As part of a change to bring special offers into consumers decision making process; create a small interaction for ‚Äúwhen would you like to book‚Äù, which lists available times and the available offers inline. Previously you had to know what existed before placing the order. I wrote the API and JS to drive this.
  • With the CEO ~ a sales specialist ~ improve quality of inventory. I performed actions to fix missing or incomplete data, so a search result may be useful to the consumer. We deleted many items that are irrelevant data from our DB. Secondly adjust database schema, so it would answer user queries faster.
  • Spend roughly a year replacing slow-code with faster-code. This is a purely technical change, but casual consumers are never going to wait 15s for a venue listing.

Why no 'users'

A 'guy on the internet' complained about the wording ‚Äúusers‚Äù vs ‚Äúconsumers‚Äù. I defend my choice, as the following:

  • Our clients will use a subset of the our site to manage content. These people are the users. Although nobody was paying attention to the actual persona documents made before I joined the company, the roles where defined then. As the documents had not been read (or at least not read and remembered); every activity got linked to 'amorphous ‚Äúusers‚Äù', which is not helpful;.
  • Most of the people on the site are people looking to eat or drink, and similar to a search engine, the site isn't their objective.
  • In my opinion, these people will not want to fill-in unnecessary forms, have very low commitment to Venue10, have no interest in a Venue10 account, will not remember passwords or logins. Maybe they spend a few minutes per week on the site (that is a fairly heavy user). To use the site as a Consumer doesn't need an account.
  • I hope we can appeal to them with serendipitous value and add value for people exploring a location they don't know.
  • They are focussed on ‚Äúconsuming‚Äù, not ‚Äúusing‚Äù
