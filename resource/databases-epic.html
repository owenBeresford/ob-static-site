Composite data

I don't get much value from ORM 1. If a query is a long statement, it will be more error prone, so should be unit-tested. If it refers to a large volume of data, there should be a strict ETL process to ensure the data is all correct and valid. Adding an abstraction may reduce the code complexity, but some ORM just seem to be a change in notation.

As a thought-experiment, I could report from data relating to two sets by the following data storage methods. Need to add a shared index / link table structure as initially they are unrelated:
`In these notes a RDB is assumed to be PG, it has good docs. Any complete DB should support all the listed features`.

  • Make new bigger table schema, import data, and use simple queries ... see BigQuery 2 3 for this
  • Simply add a few new indexes so there can be a join statement (where this works) ...
  • Simple query and the data aggregation in a server lang. Mostly [deprecated] ... maybe MongoDB 4 5 and Node 6 is a good match due to same data types.
  • Simple query + HAVING & WHERE 7 ... any complete RDB
  • Simple query with GROUP_CONCAT aggregation 8
  • Simple query with aggregation in DB ... shout out to Postgres 9 10, as this is still fast with little RAM
  • Data selection from complex indexes, so the computation needed was perform previously to make the index. Only narrow application scope. ... see Postgres smart indexes 11, any DB with complex views 12, or BSP 13 ~ this theory/ equation is used for many many data handling tasks.
  • Create RAM only / temp tables, populate them, then query against these ... TEMP TABLE 14 15
  • Create views, query them. I have less experience with high performance versions of VIEWs 16
  • Create a STORPROC in the RDB 17 18, or map reduce etc in NoSQL, eg 19. Ok, PG does 20
  • Use TRIGGERs as a smaller scope alternative 21 22
  • “single” Normal query and WITH statement as a second query. Uses time sequenced locks, so better performance 23
  • Subqueries in “single” statement, for DB that support them 24 25
  • The SQL WINDOW feature in “single” statement 26 27. I think these options should be tested on data that is structurally similar to your production data ~ its too complex to make a blind assertion on best model ~ some analysis 28.
  • Fuzzy matching via ai-vector-stores or Vector tables in an RDB 29 30 git
