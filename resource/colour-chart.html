This is the cleaned up version of a colour chart I made to talk about software quality. There is no need to learn about technology, but there is is need to learn that engineering process is not optional if you want dependable outcomes. Larger and more established companies may not need this type of description, as the project plans are less nimble. The abrupt pivoting on direction of work is not good for tidy sprints, but allows customised service demo per meeting. The very fast-paced release schedule is supported by having multiple UAT instances, and good/ updated docs about what has been deployed to where.
Yes this scale goes red to black, just like accounts P&L sheets. Companies should not even pretend that red status items are ever profit.
