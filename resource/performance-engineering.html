Intended audience: hiring decision makers, or myself if I am quoting it.

Typical activities
Most of my experience is moving companies from technical idea into startup status. I am normally adding more user journeys, more engineering maturity and stepping-in to help resolve any problems mostly caused by low current staff numbers.

My most emphasised areas are:

  • user journey ideation and basic UX analysis (small companies do not always have a budget for an extra person just to do UX, UX is not the same as UI design)
  • new technology or tech process adoption
  • knowledge retention and awareness (i.e. documentation and graphic design on various things)
  • data visualisation (at Hoptroff, this was mostly graphs and charts)
  • performance optimisation, on DB, webapps, websites and automation levels (yes this is the hard and crunchy area of my work)
Performance engineering

If you are looking to hire, you would like to know what my abilities with scaling platforms and tools are:

  • The biggest websites I was building were MotorTRAK in 2013 (luxury car brands), and probably my Seven contract in 2014, as that was used as part of Sainsbury's site.
  • The API I built at MurphX was a more highly trafficked web-service than other startups I worked at, with abit more than 300 clients and over a million active/ simultaneous users. The ISPs sites obviously weren't the primary internet destination for the user-base, but the “pre-pay topups” feature, the “monthly usage” and the “availability checker” got volumes of traffic. I normally skip this as it was my first role, and performance engineering work was entirely focussed on the DB and network stacks.
  • I could write text like a blog article with many generic tips, but I'm not sure this would help anyone.
  • However, I have built many vertical-scaling improvements, and this is similar work. If I have complex compliance report (like Hoptroff), and I improve the data handling so it uses about half the earlier RAM, and a quarter of the execution time, I have scaled the code.
  • I have organised and written data pipelines, as a data centric approach to getting data into or out of a system faster than many webpages. This involves adjusting data formats at scale asynchronously, and putting data in “pages” just like RAM management. I have some use of micro managing routine tasks, like disabling RDB indexes before a large import, but this is to be done carefully in a production system that customers are using (best example FairFX).
  • The other thing commerce needs is tight maintenance windows. I use careful planning and automation as default steps (best example Hoptroff, as I was doing everything for the web technologies).
  • .
  • I made search features at Swyvl, Hoptroff, FairFX, MotorTRAK and Venue10 better/ more useful and faster by better code and understanding of what databases are doing. I improved Hoptroff's second product search by adding a denormalised cache table. I made the search feature that Swyvl uses for most user journeys faster by not using the ORM that was applying a READ lock to every table at the same time to run the search query. My direct SQL was returning results that were more correct as a second benefit. I improved Venue10 search by changing edition of MongoDB, changing the DB driver, changing the data type in the DB for GIS structs, changing the data format and adding better code. I added frontside logic to search in a small radius first, to get some results quickly; then either get results for the next location or a larger radius if the user isn't moving.
Performance engineering process

Any non-naïve PO/PM/MD will know the areas of a product that money was spent on, and the areas that were done quickly and cheaply. Performance engineering can be delayed until a successful PoC is executed, and there are some customers; so it normally is. To-do performance engineering I :

  • Ensure I have a copy of the wireframes and user experience documents, especially if they were an appendix/ codicil/ responsibility mentioned on a contract. These also need to be the newest versions.
  • Ensure there are tests that show the expected behaviour is happening. These tests shouldn't test the HTML or CSS or JS, as “does it work as stated in the contract” isn't a technology statement. This is good practice, and ideally these already exist. Quickly resolve the failure(s), if any tests fail.
  • Work out a maximum acceptable latency for each step in the user journeys (emailed password resets can be an hour without comment, a page render must be less than 2s regardless of what it does. See Asana or Slack as examples of failing). This can often be an overlay on the wireframes. Annotate current delays on each step, to generate the order of any changes.
  • As architecture improvements will generate better results than code-fiddling, this is still focussing on “what is the solution?". List the technical sub-steps in detail that the slow steps you are optimising are doing. Big slow pages can be made into smaller pages as a performance alteration in many cases. Work out if the work could be done asynchronously to the user activity. Work out if the sub-step could be migrated to a micro-service, or a third-party; and if this would make it faster.
  • Ensure I have a test server that I can measure performance on. This needs to match the production version in the dimension that is too slow.
  • If rewriting code is still necessary, now rewrite specific code. Ensure the first thing added is logging so before and after conditions can be seen. Possibly add unit-tests here, as part of working. The unit-tests are to be focused on code.
  • Show initial tests are still passing, and performance is now passing. Run all tests, in case the slow point gets moved around due to RAM allocation changes.
  • Publish new edition.
