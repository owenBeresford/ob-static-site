Q) Why am I doing research on state libraries?
A) I have rarely chosen a state library. In most products I have improved, I start with the first iteration completed by an external contractor before I joined. I know less about choices for state libraries than many other areas of development.

Brief Intro: why use a state library (SL) / data store?

With rendering pipelines like React, there is a trend towards FP. Limitations of FP include: no persistence for data, or no predictable order of execution. You could have a persistent global variable; but that is very bad for a number of well-known reasons. The data in a SL is equivalent to a OO Components classes local variables. Using a SL creates a policy for data pipelines, for example, to make data flow in a single direction in the module hierarchy. Even with OO Components, a SL can be useful to store data for a duration greater than the Component lifetime.
If you only have a small app; or an app where all the data is known at compile time; there is less return on a SL. However most apps do not meet this small-scale orientation.
Redux strongly brands itself as a state library, the other libraries I mention do not so much. This may be due to cultural heritage from flux 1 (now legacy).

Metrics for a state library

  1. code performance of the library
  2. testability of systems built with this library (mostly: can I inject a fixture state)
  3. code complexity to use the library
  4. lifespan of data in this state
  5. number of constraints the library creates
  6. really nice but not critical: can I rewind or replay time, in terms of events happening
  7. tagging for which framework it is supposed to operate with.

Choices/ literature survey

This is not a list of every SL I can find, as this would add little value to anybody (try a search engine for long lists); but a list of libraries that are strong on at least one of the metrics.

  • react-redux npm, git mostly MIT license is a wrapper for redux to reduce integration costs. Redux describes itself as ‚ÄúPredictable state container for JavaScript apps‚Äù 2. It has context, docs or as a talk. I didn't find redux to be a terse API, but 3 claims it reduces code needed with large object hierarchies (rule 3, and improving rule 2). The boilerplate is reduced when you use redux-actions 4 5.
  • vuex nee pinia npm, git, docs, MIT license this the standard store for Vuejs. I find this quite simple, easy to inject a basic state, not noticed how long it took to copy data (rules 1, 2, 3).
  • baobab npm, git, MIT license, is an independent data store, with a metaphor. The reducers are called 'monkeys' (see git RREADME file). This has a good API for time travel (rule 6, rule 2), it only wants to store data and the exposed function are mostly pure. It uses a Cursor structure, like a DB result-set.
  • easy-peasy npm, git, docs, MIT licence is a store specifically to be easy and small (rule 3). It targets React. I'm not sure why the ‚Äúvegetarian‚Äù logo has carnivore teeth.
  • reatom npm, git, docs, MIT license; is a minimal store designed to not throw exceptions, and have good logging. Includes good debugging support, supports Mocks, DI primitives (rule 2). Supports MSIE11 and react. Doesn't impose constraints.
  • dexie npm, git, Apache2 license; is a SL that stores data in the indexDB (rule 4). If tested, I would mark it for rule 3 and rule 1. As the docs state the major benefit to using this is better error handling and error semantics over the raw browser feature. As a full SQL clone, you can access quite a large amount of different structures, and number of rows at no further complexity. As a DB, it has well defined process for concurrency. It works fine on mobile devices unless they are very old, but not Deno (NodeJS clone) environment.
