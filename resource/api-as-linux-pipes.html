Pitch/ intro

You are a knowledge worker and are accessing data from a mobile. You need to aggregate from several sources then manipulate in the mobile for presentation ~ could be local estate agents, some extra facts like insurance rates per street and a map service. It would be really nice to not have to make a client connection to each source. In theory GraphQL could be applied to this problems, but I think that requires dedicated server API, which is what this is trying to avoid.
What if it was possible to link differing server API like Linux pipes?. This is concentrating on aggregating remote API sources, not DB ~ the plumbing is different. This is a research article, I am not offering a product. As I start this article, I'm unsure if it can be delivered.
There is no logical reason why this tool ~ if it exists ~ could not be used to POST into system2 the results of GET system1. However, test first & test hard ~ see Knight Capital in 2012-08, for an example of fail conditions 1 2.
Audience
app prototype engineers, data-science modellers, data fiddlers who aren't using Pandas just now.

Limits

  • There needs to be support for flush output then exec next source ~ APPENDED sources. There needs to support ‚Äústepped writes‚Äù, so a final stream included interlaced data from each source, and interfaced in a predictable fashion ~ call this INTERLACED sources.
  • Security / auth on the resources. The linked API would need the login step.
    • Although if the sources were open-sources, this is stepped over.
  • Any solution will need to work with corp firewalls.
  • If there is need to do many small requests (ie GET /api/v2.5/things/223483423, GET /api/v2.5/things/3524663 ) buffering intelligently is necessary.
  • Buffering in general is needed as this is the internet.
  • Sensible timeout handling
    • Should be handled like a SQL transaction ROLLBACK.
  • As auto-code, this may not meet any performance requirements without effort ~ eg enriched but stale stock-data is valueless.
  • ‚ö† WARN ‚ö† Many of these options handle data-lakes frequently, it's essential to add some sort of limiters to avoid accidentally attempting to download a multi-terabyte of data onto your phone.

Possible solutions

  • Create an ‚Äúeasy UI‚Äù on top of AWS Lambda docs. Other Cloud vendors are available. Licence for ‚Äúproduction‚Äù is a commercial contract.
    • Gains: very workable, avoids problems with mobile IP. Can fairly easily correlate IDs from differing systems, as this is a full programming language. Should start with a ‚Äúgood‚Äù IP. Mature error handling can be added.
    • Losses: need to design data flow and make software for this option. The dev needs to create a quick JS/TS function. Needs AWS account setup, and per user account (assuming OSS for software). Users would need to trust the storage of authenticators.
  • ‚ÄúNot serverless‚Äù services like Vercel docs 3 can be used, but this is ‚Äúan API in a fancy hat‚Äù. Licence 4 but for ‚Äúproduction‚Äù is a commercial contract 5. SDK is git, I believe this can be run on the service.
    • Gains: more features than Lambda, on average less code needed, correct spelling of Greek words. Mature error handling can be added. Tidy workflow for static or low-context artefacts (eg puzzle game services).
    • Losses: another DSL to read. Not as simple as other solutions
  • Reporting platforms like Pyroscope + Grafana ~ this focusses on headless aggregate access, not dashboarding. docs docs these are OSS.
    • Gains: low-code platform. Pyroscope does include an API client 6. Admin for Grafana can be done via CLI 7. There are cloud packages for this service.
    • Losses: Can't find a headless option to fetch data for Grafana so FAIL.
  • Try Airflow docs licence Apache 2. There is an API client 8. A diagram of supported DB sources 9.
    • Gains: CLI access 10. Inline API client, via Python 11 ~ a few lines . Airflow can be setup on AWS or similar clouds 12 13, and also locally (as Apache foundation). A feature called DAGs offers scheduled events 14. According to 15 Airflow can be a webserver, so able to serve results.
    • Losses: Need to make a structure like a .terraform.secrets file (UPDATE IF FOUND)
  • Try Snowflake docs concepts 16, licence is commercial 17. Snowflake has many subprojects for integrations git, boast list 18. Tools to make use easier 19. Do not confuse with Snowflake.net, which is an older commercial ASP project 20 in the same area.
    • Gains: data can be accessed in other API 21 22. Has support for various data 23. Data can be manipulating in Snowflake, then GET'd 24 25. Snowflake has reasonable error handling 26 and for STORPOC 27. Docs for auth on API 28. There is a Node client project, I have no idea if this can be run in a phone npm.
    • Losses: There have been multiple security leaks with Snowflake ~ although I do not know which party failed 'due process'. There are many articles referencing Snowflake auth mechanisms, please read them.
  • GraphQL hosting vendors, Hasura or Apollo etc. Relevant notes on GraphQL itself. As an open format, there will be at least one free version. This option is more sketched as it's not very aligned.
    • Gains: Can download API with GraphQL, sample in Apollo 29. Downloads, YES. As much security as needed, YES.
    • Losses: You will need to write code [Boo-hiss] maybe YML [booooooo!!!]. This will buffer data in your phone to do transforms [Boo-hiss].
  • [CHEAT solution] There integration features (scheduled events, web ‚Äúhooks‚Äù API) on services like Stripe, but this is a narrow scope solution docs, docs.
    • Gains: strict security, high availability, good brand, little actual code required
    • Losses: only applies to mission of hosting service (payments for Stripe)
  • [CHEAT solution] when all involved are your services and on the same machine, add a ‚Äúnamed pipe‚Äù (linux) or IPC (win32) for faster comms. Example walk-through 30 (not my code). More complex shared memory options are available at extra effort.
    • Gains: this is fast, secure, and reliable (away from processes falling over)
    • Losses: same machine only

Confirmation of deployment options should be involved in solution selection.
