I am an experienced software engineer, and have built things with JS and TS depending on scale. I have used a wide range of JS frameworks; but can also write in raw/ vanilla JS. In 2022, I am refitting this article; I need to update this every three years at the slowest. My experience as a JS and TS based developer is:

  • All my code is “use strict”. Due to writing in other languages before JS, I rarely hoist;
  • I write JS to ES2020 level. I am tracking the “current stable” version for TS, and currently have v5.something installed (I check this with every new project). As of 2021 I am deleting with joy all the es5shim import statements, as MSIE has been retired by Microsoft. The current browser that needs the most compatibility support is Safari;
  • I attended a JS conference in 2020, see notes, ands also a second one in 2021 datavis one;
  • I have used Typescript at Swyvl, and for open source. I think the type system in Golang is better; but I see the commercial value in using Typescript for projects, in the browser or in a server side API;
  • I have used Node commercially, starting in 2015. I used a variety of versions of Node, but strongly favour newer versions. I have updated versions of Node several times. I have built API on cloud services, and as Lambda / Serverless API. In 2024, I am writing as JS modules for Node services, which moves Node closer to other languages;
  • Inside Node I used common tools like Express, or HTML templating libraries. I have used several ORM, and can use SQL directly. MongoDB is obviously a DB with a JS interpreter. I can use the POSIX fork() and exec() calls in Node;
  • If I was starting a fresh project, I would look at Deno for serverside JS; as it's easier to manage deps;
  • I have ideated how to link fast-to-execute languages into Node nodejs-linking, and improve JS management js-backchannel-logging;
  • I have used React to build three products (this took about three years, one was small), with v16, and v17 (across UX-engineer and reactnative-developer ). I built OSS with v18. I spent about 2 year using Vue2 to make a product (also Hoptroff); and used Vue3 for recent OSS. I worked for over a year on a product using Angularjs at lead-engineer. I use the APIs for triggering events properly, so there is no 'busy waiting' status;
  • I use npm for a JS package manager. I write complex scripts in the run scripts section. I organise to not commit result files to GIT (logging, or compiled binaries). I manage the deps to ensure they aren't too old, with third party source without any tests is expensive in this scenario;
  • I generally build PWA features, and carefully monitor browser support for various features. As many people would aggregate UX with client side work, I refer to some case studies UX and more UX, and third;
  • I prefer to write in objects/modules, it makes testing much easier and cheaper. I can write code towards OO or towards FP architectural styles. Unless I am rushed, I extract Services from Components to make testing easier / faster;
  • I use a range of test tools. I currently target testing via Jest, Vitest and Storybook. See storybook and vitest for more details. Historically I used Mocha/Chai for example a demo, and qunit. I have used Cypress briefly (Swyvl). Before I was aware of Storybook, I made a simple ux-verify library to plug into Mocha, Jest or Jasmine tests;
  • I recently updated my practice for documenting JS/TS code, please see docs-for-js-ts, with a sample in a repo Historically I had a short term hack in an IDE to create API docs. At Venue10, I wrote actual Swagger docs, but normally create locally hosted docs. I would favour Confluence as a public docs server rather than Swaggers hosting;
  • My libraries are tidily built and don't pollute the symbol-space any more than is necessary. With modern “modules” JS, I get a closure for free, and do not need to wrap my code in one anymore. I know what the self executing function declarations are, but generally only use them in jQuery;
  • Focussing on reports, I really like datepickers; as it removes two days off any isolated platform development. UPDATE: Unless I am doing something complex with date ranges, I am now using plain HTML5. Previously when I needed to support MSIE, I normally used a JS date picker and a clock picker;
  • General consensus is that data manipulation should be on the server, so it can't be disabled;
  • I have worked in high volume services/sites, but the most scalable server languages aren't JS. When I was extending Venue10's systems I added the service monitoring features, and service availability tweaks, to make it a more useful and regulated service. At Hoptroff there was a devop colleague; but I contributed scripts for his tooling, and database audit scripts, which yielded more impact. At Swyvl, I improved uptime, improved performance, added tests, added monitoring to the Typescript-based product;
  • I authored some demo code with websockets, but as fundamentally it's a TCP socket, this is no different to normal POSIX programming. I have written a lot of normal TCP and UDP based programming;
  • With regards to UI, I target HTML5, mobile and RWD / PWA. I use CSS frameworks for cost reasons. My personal site is targeting desktop mostly, due to the volume of words; but it includes CSS for mobile;
