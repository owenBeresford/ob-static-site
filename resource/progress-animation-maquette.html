Context

A popular UI interaction is the progress marker. Thinking from a usability perspective, I do not think they are a good feature. Everyone would prefer a fast system to one that says ‚Äúplease hold‚Äù all the time; and the animation foregrounds the fact that they are waiting. If you have an average speed system, there may be a short blip while something loads, but a ‚Äúplease hold‚Äù notification for 0.5-1s is disruptive to concentration more than supporting.

Technical Notes

The legacy-tech hourglass symbol
If you are designing the UI in a big system, careful use of caching and multi step API can minimise the delays, so there are not that may of them. A browser on a recent phone or laptop should be able to do most things with little delay. The most common delays on websites are the negotiation process for displaying adverts, and downloading large frameworks/ images/ sound files. The ad servers are generally fast, but if there is a bidding process between 20-50 different companies for who wants that user's impression the most, this is iterative process and why it is slower. None of my employers have gotten revenue from advertising, and it seems bad UX to force users to wait for the adverts that they do not want, YMMV. For the assets mal-feature, taking Asana as a case study: they have little 3rd party advertising. When you login, the browser blocks for 5-30s to start the interactive GUI. The delay is greater on slow internet machines, or machines with little RAM.
Writing as a UX centric person, if using a ORM slows down your UI, so you need progress indicators; just use/ learn SQL, its pretty much a functional language.

Some people use the progress-bar effect vertically, as a scroll-bar presentation; I have most often seen this in forms. This article is not about this usage.

Maquette

There is a guide to build a progress widget manually 11, which I'm not following. Use the first version as a metronome, they all have a 2s loop time. I think some of these would look better with a different speed.
With an adaptive interface, delays could be shown with changing colours, if that suits some users better. Also the loop time can be controlled from a central config. All these widgets are just CSS, they should have no impact on any features of an App (no extra downloads, no JS threads, no blocking code or events).
As this is a maquette, all the code is in this file, for better visibility and portability. Some handy references 12 13 14 15 16

Note: I am just using a circle, as it is easy and light/ few bytes to download. The animation could be applied to a logo if the preferred visual was a tradition hourglass. For production use, these should also set the mouse cursor.
