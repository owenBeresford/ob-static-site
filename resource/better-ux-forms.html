I was thinking about some bad forms I had used recently. So I thought I should define what is a “good form”.

Form interactions as an experience

The references documents for this tend to be long, so would hit multiple ideas. I do have a decades worth of experience, but I think a published reference is a better touch point. During 2017, I read a huge (file size) ebook, that showed UX changes by an UX agency across their clients' sites. The ebook was too big for me to save it on my phone. I will structure it as a list of references at the end.

  • It is widely known that “screens” / multi-step forms perform better than single large scrolling forms. I would love to be able to apply this via CSS, but injecting a Form component based on screen size isn't what CSS is for.
  • It's 2024 today, I really hope I don't need to mention making mobile the first device to support.
  • Remember previous input if that can help, but not for passwords. Remember it is less time to edit a nearly right address than to re-enter the entire thing. In many places it's convenient to set a useful value as the default (e.g. “enter country”), but in others it's counter productive (e.g. “carpark location”).
  • Automate sections that can be done, like PAF. Use device hardware if it can help (this mostly applied to phones), For predictable input, either ask for it in a low effort style ~ like checkboxes, or add autocomplete. Automate “manual work” like capitalising postcodes ~ but not for passwords. It is more than twice as important to automate the manual work on mobile.
  • Use structures so users are not forced to get involved in questions they don't need to answer.
  • State average time to complete form at start. Be clear about the progress through the form. Be clear about goals, and motivations for filling in the form. There have been reported usability problems when users didn't notice that a form changed “screen” and so the user hit back as a default action. Marketeers are told to keep using the word “free”, but this doesn't apply in all situations.
  • For low input forms, there is research that reducing eye motion improves “form filling performance”. If the user is entering a line of text on each form item, this may not apply.
  • Use inputs where the entire range of possibilities can be casually seen. This implies that checkboxes are better than open SELECTs, and open SELECTs are still better than closed SELECTs. Also have grouping structure on long SELECT boxes.
  • Apply “common tropes” e.g. the credit card numbers are in groups of digits. This reduces the mental load to fill in the form. Supply relevant size to the form item. Use placeholders to guide users if there could be more than one style for data entered (e.g. when asking for phone numbers, offer a suggestion, so users can see about prefixes, and number grouping).
  • Avoid unnecessary questions, people aren't filling in forms for fun. Structure the information so it makes sense, maybe talk through your form as though there was no devices, and you needed to voice call people to get the data. Consider whether optional fields are worth displaying at all. Use a “Thanks” page after submittal but ideally somehow that doesn't make the user do an extra interaction.
  • Unfortunately colour shouldn't be used as sole sigil, due to fairly common colour blindness.
  • Supply validation feedback as early / fast as possible ~ I always validate on the client side, but then again on the server. The error messages upon failure should be specific, and actionable.
  • General advice on text sizes applies here very strongly. Try readable-fonts
  • For sales forms, labelling in important: make all calls to action (CTA) very clear and personalised in terms of wording. Also high contrast to backgrounds.
  • Think about adding a CAPTCHA service, this first link is offering alternatives 1. There are many CAPTCHA vendors (this list is just a few alternatives) 2 3 4 5 6

Negative/ dark UX patterns

Most of my site is about technical ability, the following structures are common sales techniques. Using those structures will make your employer the same as its competitors. I am not endorsing dark patterns, just realistic, every normal person will have met these in general life before they are old enough to be a qualified developer ~ “boot camp” tech courses use most of these patterns. In the US they trust in a god, in the EU we have GDPR.

  • The simplest dark patterns are really traditional, and used pre-internet. Adding hidden costs to a consumer transaction. Sales misdirection, likely to support the hidden costs. Making fake social proofs ~ like hiring a celeb to endorse your product, even though they would've never actually used it.
  • There are traditional revenue boosting mechanisms: 'bait and switch' on the product that the user is looking at. In today's terms, this is encouraging users to start to use a freemium edition that doesn't actually do anything, so they they will make the mental commitment to buying the normal product. Or time limited demo editions that will auto-enroll the user to the paid service. This is why users should be cautious about supplying card details for a demo. Deceptively adding extra items to an invoice, like a subscription, an insurance or fast delivery.
  • Non-conspicuous adverts have been around for a long time, look at most elections as an example. On the internet this can be a 'advertorial' or an 'infomercial' formats, where the media subtly pitches their product hoping the user doesn't notice.
  • Historic US sales men got a reputation for some types of activity (note they were nearly all male at that point). Fast talking, trick questions and misdirection. 'Confirmshaming' ~ which I'm sure at least one religion uses ~ framing choices so the only one that looks neutral is the one the business wants to happen. In the US, preselected sales options were common, GDPR stopped them being so obvious.
  • Processes that make it really easy to signup, and hard to cancel are well known. This is known as a 'Roach motel'.
  • The newer edition of social proof is getting a user's friends (mostly the people listed in the address book or friends list) to look at an advert for the service in front of the user. Linkedin was an early adopter of this, and got quite famous about using it.
  • Structuring the sales funnel so it is hard to compare prices, so users will over-buy. This is also applied to stopping external price comparison tools from working. Traditionally labelled as “Price available on application”. Building the funnel to have a strong FOMO ~ a fake alert saying “only a few of these left, buy now!".

For extra study, look at usability and UX reports, which have a more focussed writing even if nothing else is interesting to you. Nielsen Norman group are a well known example, eg 7.

Are Dark patterns useful?
Many businesses expand their web platforms by imitating a competitor, as they observe the techniques were working. Linkedin used the “spam your address book” option for years. I suspect it netted them more accounts. If the motto of your company isn't “do no evil”, and it is inside the law, trade is trade. It should be possible to make a sales funnel that is clear about prices, and mostly transparent etc etc.
If there is a bidding UI, supplying the user with fresh information about availability and contention is a good feature. The early versions of ebay did this dynamic feature, and it wasn't thought of as a dark pattern. Now every Shopify site does the same interactions, and it is on these sites. Mostly as Shopify sites are not competitive bidding.

Idea: Form design without JS

As an experiment, if a form was built with only CSS and HTML5 it would probably be faster loading than a JS monstrosity. The number of things that can be achieved with CSS is increasing, and clearly we all want the smallest, fastest-loading approach to be used. This list of ideas is also presented as an oversized code-demo available as form-no-js-maquette.

  • A short list of hopefully well-known HTML attributes 1
    • “required” attrib,
    • judicious use of “autocomplete” attrib 2. I'd like to be able to offer auto complete values that the server knows about and that this client (browser) doesn't. Unfortunately, I can't see how-to do this. This “made by vendor” auto complete is NOT relevant to personal info,
    • pattern attrib to set a regex that the browser will run automatically,
    • “min-length”, “max-length” attribs for number of characters on text input,
    • “max-length”, “min-length” on numeric types
    • 'input type=“date”', 'input type=“url”' are now supported rather than being theories,
  • Your framework should set the next items for buttons, but check its present 3 4.
    • ` touch-action: manipulation; user-select: none; `
  • If your framework has any points for accessibility should set this selector 5
    • ` input:focus-visible `
  • A CSS selector for non-empty form items that currently invalid. Use this structure to not whine about form items that people haven't touched yet.
    • ` input:not( :placeholder-shown ):invalid `
  • A CSS selector to display the help text at relevant time, mostly the suggested use is password rules 6
    • input:focus + .help
    • UPDATE after dev time, the following works better with modern browsers
    • ` input:focus-within +.help`
    • IMPORTANT: if some scripting is necessary (like email address validation), and this is blocking the form, there should be a message explaining the block state to the user.
  • A useful CSS selector to display that a form item has a good / accepted value (add a tick or similar via content).
    • ` input:valid:after `
  • A CSS selector for non-empty form items that are currently invalid. Use this structure to not whine about form items that people haven't touched yet 7.
    • input:not( [value=""] ):invalid
    • ` input:not( :placeholder-shown ):invalid `
  • A mostly CSS widget for “other option”, use a SELECT for known items, have an OPTION labelled “other”, when “other” is selected, show a text input 8.
    • ` body { --extraInput:“none”; } `
    • ` select.withOther option[value=“other”]:selected { --extraInput:“block”; } `
    • ` input.withOther, label.withOther { display:var(--extraInput); } `
    • DISFUNCTIONAL: CSS isn't allowed to access or modify user input for security reasons.
    • “Paranoia first development” says to add and remove a “disabled” flag on the text input, but I can't do that via CSS 9. The disabled flag would stop the input being sent to the server, so allowing the server logic to be simpler and hopefully with less errors.
  • CSS selector to style-up form uploads 10 11
    • ` input[type=“file”]::file-selector-button `
    • Please set input[type=file].accept it means the UI is more informative
    • I think JS to allow drag and drop files is acceptable, if its not the exclusive access path . Also, the demo at the end of 12 is nice for desktop users.
  • A CSS based form blocker, to be in effect until everything is filled in. In practice the plain HTML attributes should deliver this effect, but CSS extras allows branding to be applied 13 14, the control on the submit button was innovated via 15
    • ` body { --formEnabled:“notAForm”; } `
    • ` form > :not(input:invalid) { --formEnabled=“Save”; } `
    • Adjust the text above to match your form
    • ` form input.submit[value=var( --formEnabled )] `
    • UPDATE, I could use the above approach, but I can also use
    • ` form:has(input:placeholder-shown, input:invalid) input[type=“submit”] `
    • and disable mouse interaction with `pointer-events:none `
  • I have looked but been unable to build a formatting instruction to apply gaps into text for credit card numbers. Obviously this can be done via JS, probably standard widgets .
    • A traditional solution (i.e. 2005 era) is to use multiple text inputs, but then there needs to be a thing to automatically jump to the next input ~ JS.
    • TEST THIS. A list of CSS features that I can't currently apply to solve the problem 16 17 18 19
  • A selector to display a card vendor logo, based on the first 3 digits of the card.
    • ` form input.credit[value^=“44”]:before { content:“Visa "; } `
    • DISFUNCTIONAL: CSS isn't allowed to access or modify user input for security reasons.
    • Add further card vendors, and add further details to style up the card logo, paying attention to its width. A nice site that lists banks that are on the credit card exchange 20, a list of vendors 'in English' 21. A list of US vendors 22 23, CC are most common in the US, and they invented them. The second link describes the Luln verification, but I don't think that is practical in CSS. Ok, I could 24, but I think that is misplaced effort.
  • Show which form items are essential on the label not the input ~ just a readability tweak. This can be achieved via many methods, and the following are examples that should apply a extra content :
    • `form label:has( input[required] ):before `
    • OR `form label:before + input[required] `
    • A minor boost would be to disable this when the form element is off-screen, but that is left as an exercise for the user.
