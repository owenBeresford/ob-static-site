Basic HTML5 support

  • I try to have a policy that a image that is content (likely a diagram, or company logo) is present in the DOM, as I want screen readers and printers to see it. If an image is similar to “bald man #4 holding a cup, smiling broadly over shoulder”, then it should be done via the CSS and absent in the print.css or 3g-mobile.css
  • the ancient IMG tag supports “alt” and “title” labelling. Title is used for a tooltip.
  • the PICTURE tag de facto supports title (not according to the spec 1 but common browsers support having it), but requires an IMG present 2 3
  • the OBJECT tag doesn't support title itself, but can be wrapped.
  • the FIGCAPTION roughly speaking is a title element 4.

Notes for tooltips

I think websites in US English reserve 'tooltip' for inline help and image description; and not alt-text. General guides for tooltips are (notes taken from my experience and 5 6 7 ):

  • Clear but short text so it can be quickly scanned, but not break context/ immersion. Timing on when to display and when to hide messages is important to user experience; and unfortunately often different to each user (reference myself watching people use my interfaces); tying the visibility to mouse events gives users control to suit their needs; and doesn't work on mobile.
  • Highlight happy-path user journeys and goals, especially for new accounts/ installs.
  • Have very targeted text, ideally changed to match current software state (e.g. the first time you hit “save” on a platform, it has different text to later times you click it, and the first time for a new file, it's different to routine saves, for example).
  • Do not obstruct user progress, and the UI should show user progress. I think this is generic UI design rather than about tooltips.
  • UPDATE: I have been told that excess use of Emoji in image descriptions leads to poor user experience with screen readers, as screen readers understand them to be letters in a different alphabet. If it reads out a phrase that explains the letter that would be particularly poor experience, with repeated emoji.
  • Since I wrote this article, another webauthors notes “The Literary Style of Alt-Text” 8, this author is better with human languages than me, which is why I include the link. There is an accessibility awareness account on Mastodon, they cite “be mindful and respectful of attention spans. Focus on what's truly important” 9.

Imported notes for captions

This list was written by a transcription author Catlin LeFey 10, I can't map to a linkedin account; but it seems he has experience doing this work. I think he is referring to FIGCAPTION descriptions, is lightly edited for punctuation, but direct quote:

  1. Any words are better than nothing.
  2. You don't need to say it's “a picture of…" screen readers will already say it's an image.
    1. The reason not to start with the words “image” or “picture” specifically is because the screen reader will announce it with those words, so it will sound like “image image of a cat." Not a big deal, just sounds a little odd. Instead, I suggest starting with the format (photograph, portrait, chart, Facebook screenshot, text, all of these are good!)
  3. Start with the framing or format (i.e. close up, landscape, meme, text).
  4. Think about the reason you're posting the pic and describe that first, add background details if you have time.
  5. Pretend you're talking to someone on the phone and want to tell them about this cool thing you're looking at.
  6. Transcribe any and all text in the image, even if it's the only thing you do.
  7. If you've described the image in your post, you don't need to copy and paste it again in the caption. But again, don't leave it blank, just put something like “as described."
  8. You can add small subjective notes, but don't give too much interpretation of the image in your own opinion.
  9. Caption jokes are fun, as long as they still describe the image objectively.
  10. Use punctuation, and capitalize words properly. A lot of us have interacted with this tech when calling customer service or talking to Siri, so keep in mind that you're writing for a computer to read, and it needs all the help it can get.
    1. When a screen reader comes across a comma or period, it takes a little pause, to mimic natural speech. So just put commas and periods where you would naturally pause when speaking. And if you, like, use too many, it's better than not using enough. Capitalize after a period, then use caps sparingly. Capitals are read out letter by letter, so ask yourself if that's how you want them read.
    2. Probably A. Definitely not B. This is tough because every screen reader handles punctuation differently. Most will just pause at any and all punctuation without saying the symbol out loud. So “this," *this,* and ,this, are all going to be read the same way, with a slight pause before and after. It's not ideal, but it's the only way to show that kind of emphasis.

It would help if there were a standardized way to describe things. For whatever purpose.

The problem with writing for a general audience and talking to a specific person is that these are very different skills.
I've found that while I'm OK at describing cats and uncluttered pictures WITHOUT people in them, that I can do it.
But as soon as it comes to a GIF, or a meme I am unfamiliar with, or a modified picture that I can't do it

Other Caption

As a design principal “Give all users access to the same information” 11, and recommends setting an alt text of "" on decorative images i.e. “bald man #38, décolleté, with coffee + smile”; so various tools will ignore the image. It is important that functional icons (e.g. a buy button) have a alt description, as you want everyone regardless of technology preferences to be able to implement that user journey. According to 12, using FIGURE and FIGCAPTION is a good idea; as it makes the placement of the image clear in your data. It suggests that alt and titles texts are between 8-80 chars long; but if a longer description is needed to be clear, then definitely use a FIGCAPTION. There are advanced captioning options in the HTML specs 13 (see end), to allow multiple elements inside a caption; and tag different activities to each element. CAPTION is a tag option for tables and supports the same basic objectives as FIGCAPTION 14.
