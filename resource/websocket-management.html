Backfill notes on this protocol. I'm fairly sure every dev after intern status has used web-sockets, but it is often treated in a less mature engineering fashion.

Intro / Assorted facts

  • Devs with languages skills beyond JS will recognise a TCP socket.
  • It is not real-time, unlike various adverts, although it is a much faster turn-round time than wordpress. UDP for a simpler so faster solution than HTTP networking 1.
  • This has RFC RFC6455 2, RFC8441, RFC7977, RFC9220 and a Wiki 3. The wiki page can be used for an overview ~ I don't think repeating these words would help.
  • Websockets 4 have been a widely available technology since 2014 5 API for client-side Object 6
  • An alternative idea from Apple is live-streaming 7, seems to be for video streams. If you want Websockets but for large data, and are OK with Chrome only, try a Stream version 8. In the past there was HTTP PUSH in HTTP2, but that is heavily deprecated in most software, as the idea didn't work. There is a WebWorker push feature 9, but it doesn't mention how to create these on this link (I hope it's not the previous link).
  • Websockets are a 1-to-1 protocol, for 1-to-many / broadcast look at Server side events 10 11.
  • As a data protocol, rather than text document transaction protocol, there are no document fragments (mentioned in first Wiki).

Solution characteristics

  • Stereotypically for time sensitive data, where a full webpage doesn't help.
  • Websockets do not help with TCP performance, the physical network or physical distance from a server. They do side-step everything in HTTP, see also HTTP streaming 12
  • For time-limited, or latency-sensitive data, HTTP caching, web SEO, and HTTP search tech are irrelevant.
  • For traffic with many small transactions, a different protocol to HTTP may be faster. A websocket is an alternative.
  • For data where the number of possible copies needs to be limited, HTTP will need special handling, or not to be used. A websocket is an over-engineered alternative.
  • Note that by the time Websockets matured, HTTP2 was mature, they offer similar outcomes 13, BUT HTTP2 applies to your whole site (and should be used for other reasons).
  • IMPORTANT: a Websocket doesn't follow the 2 HTTP connections per domain, so allows faster flow of information to browsers 14. HTTP is also limited to 6 IPs concurrently.
  • Due to no server sockets involved, Websockets will exhaust available IP ports on servers faster than HTTP1 does. If a Websocket isn't left open (ie being used for a chat client), this is similar to HTTP2.

Use cases

  • Prompt Chat clients
  • Competitive bidding API on something
  • Games with real-time and multi-player interactions ~ use traffic habits that mix both the above items
  • Collaborative editing / multiple author tools
  • Data that services wish to avoid being found in HTTP caching services / search engines
  • Data media streaming, AKA webRTC 15 16 webrtc-test

Scaling

  • Obviously all generic scaling tools like front-side cache, etc will apply to an architecture that also uses Websockets. See 17 for more details.
  • According to 18, on average, a HTTP2 transaction has a 2,000 byte overhead. A Websocket has 2 bytes. Particularly for many small transactions ~ like a single real-time chat message ~ this is a visible percentage.
  • According to 19, a range of concurrent Websockets per server are guesstimated. For sensible use, reference 5,000 ‚Äì 20,000 end-users, concurrently.
  • Assuming Node as the server-side language, the memory handling is a bit fiddly ~ see 20 or it has a leak.
  • Assuming the work is using Node, it's recommended to use ws or tsws, rather than sockjs. The average consensus is that extra features of sockjs do not deliver value eg 21 but I read this multiple times.
  • A Linux server is better optimised for high traffic network operations than win32 machines. Also the Linux server should be tweaked to support larger numbers of file-handles and open ports 22 23, a longer theory and data 24.
  • Obviously communication via shorter messages can happen faster than longer ones. If the messages in the Websockets are encoded via Protobuf git, this should save bytes. If each message is just a packet type and a fairly short string, this step doesn't add any value.
  • When you need concurrent communication with >10,000 parties, it's important to look at ‚Äúthe first mile‚Äù of networking, or the traffic will be queued up ‚Äúat your front door‚Äù (ie near to the server in networking terms). This is out of scope for this article. The consideration will be short if your service lives in a retail Cloud. An article with nice diagrams for scaling 25.
  • A RnD project to operate 1,000,000 Websockets in a Node process git. This is scaled via Docker to support the necessary number of IP ports, and RAM via Clouds machines. NOTE: A cunning person with the root account could load multiple IP interfaces on a single ETH card, and sidestep the efficiency loss of a whole VM. Also add multiple network cards.
  • Contriwise to above steps, large scale problems (ie 10,000,000 connections) should be in a single process outside of the kernel where possible 26 ~ this is also what Oracle inc decided in the '90s.
  • ASIDE Some articles describe connection pools eg git 27, but as each event has a directly assigned event handle, I do not see how this works. In lower level languages, a connection pool is to reduce multiple event polling to a single loop and poll() call, so save CPU time. If each event handler is separate, nothing is gained.
  • I think this list should be longer, but context free optimisations are valueless.

Test handling

  • A test-server for a known running URL, to test that libraries are installed correctly ` WSS://ws.postman-echo.com/raw `. As an example for generic ‚Äúnot working‚Äù, use the variant without SSL, ie ` WS:// `. There is GUI to look at transactions if this helps (ie HTTPS://).
  • Test clients that support Websockets 1, 2 ~ more common than I was expecting. Probably the easiest integration is to embed `wscat` docs git npm in your normal unit-test process (when JS).
  • Assuming complex behaviour by client side software (maybe an agent?), so there needs to be a test server to show he client behaves as expected. I am unable find a ‚Äútest server for websockets‚Äù library. It should be affordable to make a static server that replays scripts via a Websocket (think `expect`, but outputting via the websocket)
  • Unit tests should be constructed to show requests and responses for the Websocket API, ideally isolated from the networking layer, so the test runs faster ~ using your normal test library setup.
  • The ` tshark ` packet sniffer can be setup as 3 to show traffic. For HTTPS (likekly), follow 4 5. For in-browser ‚Äúfirebug‚Äù debugging, this Chrome plugin looks useful 6 7. This concept is strong as you can capture HTTP and websocket traffic concurrently.
  • A liveness test is easy as long a your languages supports WSS, see `wscat` above as a reference (easy from Node).
  • Performance tests should be created, looking at concurrency and transaction time. Example, a service using overly-simple architecture and Node that is only fast with one client, has no value. This could be easily created by ChatGPT etc.
  • For a ‚Äúnormal webservice‚Äù it is essential to test with over a 10,000 connections, as the socket is permanently in use per client. Basic implementations will fail (note 10K). There should be some test runs that report performance across the normal internet, as this is what your users will see and do.
  • As a standalone URL tester, cURL can be extended to support Websockets. See notes below

Web GUI tools outside of firewalls, that are not linked to your test platform are of little value for unit-tests. They can be used for exploration / debugging, assuming a publicly accessible copy of the work-under-test is present. I would suggest a copy of `wscat` unless the work is mature enough to be accessible.
