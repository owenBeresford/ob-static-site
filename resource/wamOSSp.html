What opensource

When talking with a non-colleague about one of their client's current system, I noticed what I say I do (in terms of architecture and process); and what my open source list show contradict each other. That I can install wordpress, and write an article on how to install wordpress is not worth doing. This is a list of where I invested my spare time:

  • my OLDEST web-renderer;
    • A long time ago I needed PHP5 experience; but my then CTO didn't want the risk of upgrading operational platforms.
    • First version was about 10 days' work between contracts. My biggest test population in a single project in 2010.
    • Started in 2009; platform version 2 was 2013; the source in github is build 3 from 2016, and much better managed; build 4 was skin only, and 2018; build 5 is JAMstack and newer phone support. Build 6 improves accessibility and improves loading time, this is what you see now.
    • For features pls look at the OLD web-renderer (in github as icelinerenderer)
  • my static site project ob-static-site;
    • where all the content and front-end work is.
    • Probably this counts as version 7 skin by now. Current edition is HTML5, accessible, fairly adaptive. The next project makes it more light weight.
  • replacement JS (extra-panda);
    • I replaced a large volume of old code that was hard to maintain and update with a new small fast and well tested code.
    • modular API is much easier to manage. New code has 4% volume of old code.
    • Good test coverage, TS, Vitest, 2024 work across 2 months.
  • cat profile demo;
    • A simple demo to deliver a solution for React component testing that had low overheads.
    • Has been used as a tool setup repo (JSDoc, Vitest, Storybook)
    • CODE has expired, but was built during 2022 I think
  • punit;
    • I wanted my phpunit skelgen tooling in Perl.
    • ? 2015 ?
  • phpunit-skelgen ;
    • I did maintenance on a older project that the owner had shelved; I made it PHP7 and made the result unit-tests easer to use.
    • ? 2016 ?
  • sums + mocha demo in JS;
    • I wanted to have a reference demo for setting up Mocha.
    • The Mocha bit was done in a few evenings. It took ages to get my trees to balance properly, so operator precedence worked correctly. I think 2018
    • I deprecate Mocha usage, by the time this was complete.
  • A minimal customer-managed CMS (was accessible historically as the TBBS website);
    • I needed experience in Agile development, and YAGNI approach.
    • DEPRECATED 2011 delivered in 2 months, end to end; work (and meetings) only at weekends.
  • presentation asserts;
    • A JS library to inject into unit test frameworks, to improve ability to make declarative statements about the current DOM. Returns CSS units in normalised form, so unit tests are more reliable. It reduces code length in the unit-tests.
    • Code is in the conf project in my git account, also links a Color library for normalising color formats.
    • Done in 2019 I think
  • Vitest extension (not yet released);
    • A short term feature to allow vitest scripts to be run inside a browser, to test HTTPS only JS features and get better CSS evaluation.
    • Outputs a standard unit test PASS/ FAIL note to the console, just like Vitest, Jest etc.
    • Released version will be in TS, and includes a build step to bundle the unit-tests scripts, so the test server is simple and needs no maintenance.
  • My current shopping app.
    • I wanted more Typescript experience.
    • Done in 2023, shelved

Why testing

On my personal github repo, there are 18, UPDATE: no 12 projects (one of which is an aggregate); and source for three test frameworks in three different languages and four test templates/ libraries; on my website there are loads of articles for test technologies. I deleted alot of old JS that was wasn't managed. Why all this attention on testing?

Reasons

  • I mostly work in startups, there isn't enough daytime budget to do “nice to have” items; have to MoveFastAndBreakStuff (not affiliate link);
  • I do use tools built by someone else where possible, as its cheaper; these are the ones I can't find (or last version isn't maintained);
  • Most people do not think test tech as being a trade secret or high value IP; so they see no conflict of interest with it slopping around between projects;
  • Being abit dyslexic, there are times when I make ridiculous typo-style errors, tests avoid these going anywhere;
  • Test infra/automation is mostly immune to fashion; so if I make a tool now, it will still be useful in 5Y time;
  • I like being able to deliver a fairly high level of test coverage at a low cost. Managing the tool-chain is required, and writing it helps a large margin (referencing the tweaks to phpunit-skelgen).
  • Some of my test solutions are really creative, er, solutions.
