Intro

For large numbers of developers on mixed machines, it is smart to execute lint / prettier / tests / compile automatically. For a useful amount of CI/CD automation is necessary 1, and adding the activity in the Git service is parsimonious.

I didn't write this article about 10Y ago, as I thought at the time that it was very simple, and I try not to copy-and-paste the internet. Things have migrated from a service you install and setup with YML on your laptop (like Jenkins) to a service you setup with YML on your code repo (see chart). This makes coordination between team members easier. I have read the included references, although for lower friction use a YML validator frequently. The inputs for the automation services are not shown to be complex in these blogs. I have included the official manuals in-case the blogs omitted important steps.

Local step for Git

Git the locally-installed tool can run scripts on ‚Äúversion control events‚Äù 2 3. These are not sent to the server on Git-push, and I think make most sense for ‚Äúlocal weirdness management‚Äù. The first link lists the hooks that exist (about 20). As they are running on your local device, they do not cost anything or need any SaaS accounts.
This is useful if you run a different OS to the rest of the dev population, and want different line ending or char-encoding, just like the legacy FTP 'ascii mode' 4.
No code supplied, please see the first two links. I think running a list as a pre-commit is an efficiency measure if your IDE doesn't perform this. This needs to be aligned with a no-warnings lint policy, or it is obstructive paperwork. This policy is disruptive if abruptly applied to a existing project where dev release speed was prioritised. Probably the smartest process is to increment the number of active lint rules over months (much longer than the average branch life span). This means every branch may change a few details, but each step is small in time and is easy to read as a `diff`. Assuming a steady size team of devs, there isn't likely to be a performance gain by adding `eslint` rules. Sometimes weird compile messages can be explained via lint, but hopefully that is a rare event. Needing to failover to the external tools manually means your IDE isn't setup properly.

I'm not sure if Terraform should be strictly classed as a local step. The Terraform daemon does run locally in many cases, but for realistic service monitoring, it would be run on a server inside the production firewalls. I made some notes terraform. This would only make changes when asked, and does support diagnostic/ review modes. Terraform can be triggered from other services e.g. before black-friday sale begins, double server allocation on cloud host. #TODO Add Jenkins or other locally hosted tools.

Note: with auto-deployments, the entire process needs to be automated, as you are not looking at the source code during these steps. Yes stupid-YML-error, I'm looking at you.
