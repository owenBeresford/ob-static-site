A very boring list, but necessary to have a check-list. This article seems the simplest solution. At work previously I have changed server-side language 3 times in 2 months, and my tooling in each case was different, with different levels of maturity.
As a ‚Äúafter a few years‚Äù review, the easy-to-find advice is either badly audited, or incrementally increases. I have a domestic document on this area from 2013, but that is far too old.

An accessible internet-facing website should have

Artifacts

  • Artifact: 'contact-us'. Yes this ties up staff, selling without communication isn't a viable process.
  • Artifact: To comply to bits of legislation in the UK and EU, the site should declare & manage cookie usage. Nearly all of these that I have seen are a third party tool.
  • Artifact: a sensible domain. Whether it's best to use a ‚Äúnew TLD‚Äù domain, or a more traditional '.com' will vary with your product offering, and who your clients are.
  • Artifact: terms & conditions (for anything commercial).
  • Artifact: a sitemap, for search-engines, and less traditional users (e.g. blind people).
  • Artifact: for commercial ‚Äúpremise‚Äù, an anti slavery statement is useful (UK only). Advice here 1 2 3.
  • Artifact: cybersecurity regulatory framework document, UK only 4 5.
  • Artifact: consider if logins are actually needed. no login= no data stored = simpler life. Marketing people want everyone to identify themselves at every step so they are easier to sell to. If logins are needed, consider outsourcing the process; it's not a USP or significant user-journey for nearly every business.
    • Extension: assuming you do need an overt identification, then could it be a SSO (ie via a social media login ~ SML ?) 6. For activities like banking were a SML is not recommended, there should be 2FA 7, and consider what your second factor is with respect to ease of forgery. I am critical of every bank that limits access constraints to a 4 or 6 digit number ~ use 15+ digits and allow a wider range of characters. Remember ‚ÄúCorrect Horse Battery Staple‚Äù as an example of an authenticator that is easier for English-speaking-humans than machines 8. As a human-centric checksum, choosing passwords that you could transmit via voice comms means humans can manipulate it more reliably, but please do not transmit your authenticator. Using multiple words means dictionary-based attacks become poor again. This supports the above cybersecurity notes.
    • I think it should be standard practice to include a non-US ASCII-127 letter in your passphrases ~ so '¬£' in the UK ~ as this localises the range of people who can can type this. Not that hacking tools need to type. Note hotels whilst on holiday may not have the expected keyboard layouts.
  • Artifact: role-based access-control (RBAC) is recommended, as it allows more tuned user journeys to user needs. This supports the above cybersecurity notes.
  • Artifact: If the site is a sales platform / landing->lead funnel, add standard SEO features (I don't normally make these manually; stereotypically use a wordpress plugin).
  • Artifact: Accessibility notes, please browse requirements in accessibility-testing
  • Artifact: social media channels, communication doesn't need to be limited to the oldest technology. Your support team / sales team / accounts team who do customer interactions should be able to do this via SM channels also. Prefer direct messages, so your customers wont publish email / bankaccount etc to the internet.
  • Suggested Artifact, not required but makes communication more clear: copyright statement, mentioning owner, date, and ideally a form of contact.
  • Cookie terms and privacy statement. This item is mostly for the UK and EU. Blah, your staff will already have an opinion here...
  • IDEA, Transitional artifact, IDEA used as a cue that the page is still loading, but can be interacted with, I think it's nice to have a temp banner message ‚Äúloading assets‚Äù or similar. It is more correct that the App useful status is achieved faster rather than annotated, but the message is useful for automated testing. If a CSS content is set with font declarations, then suppressed after images, fonts and JS advertising plugins have executed, then there is better clarity on quality. Important to note: do not block users while assets are initialising.

Tech features

  • Tech: HTTPS access, and headers to force HTTPS, [1] and also see HSTS below
    • This applies twice as hard to internal communications. If there is a precise SSL setup to talk to your DB, but only 3 machines are allowed to do this, its a quality filter. Allowing elderly copies of Android to access your systems allows better customer service, but allowing the same to your DB instance is nonsense.
  • Tech: using HTML5 properly, pls see recently added tags if unaware 1 or all tags 2.
    • HTM5 is less well supported for MSIE, but we do not use that anymore.
  • Tech: Content-Type widely used header. Mentioned here as the charset clause should be present to limit an escalation attack e.g. Content-Type: text/html; charset=UTF-8.
  • Tech: a HTTP 404 handler [2]
  • Tech: a HTTP 5* handler [2]
  • Tech: a robots.txt 3 4
    • Recommend blocking search engine access to any URLs which are not content centric (e.g. contact-me)
    • supported browsers: approximately all.
  • Tech: if you use a cheap domain host, Google sometimes gets confused about domain details. Add a canonical link for index / home pages to de-confuse it 5 6
  • Tech: a Content type options header.
    • supported browsers: 93.21% nearly all 7 Absent from opera and minority browsers.
  • Tech: for branding, a favicon.XXX image; note there are a range of formats that can be supported. Pls see favicon-notes
    • supported browsers: A standard setup is 3-4 file formats, supplying a very good coverage. Single file formats have something like 80% 8
  • Tech: Asset compression [3], the process of making all the files as small as possible, so they will download faster. Very important when 2g and 3g was how ‚Äúfashionable sales people‚Äù used the internet.
    • supported browsers: all, compressed assets follow the same rules as non-compressed ones
  • Tech: HTTP2 and to upgrade to HTTP2.
    • supported browsers: 97.13% nearly all, outliers are [4]
  • Tech: Cache-Control 9 and Expires 10
    • supported browsers: assume all, not listed in caniuse.com
    • http-cache-breaker
  • Tech: relevant flavours of rich snippets / meta data, to improve accessibility to your data by both users and search engines. Again, this is too big for a bullet point, see webpage-micro-data
    • supported browsers: broad as this is a decade old technology, see link for more details.
  • Tech: Analytics, if it's inside your user-cases. When websites are sales pipelines, this is essential. I previously wrote current-gA - 2018 or for in-house analytics analytics-2
    • supported browsers: all
  • Tech: asset headers (like MIME for JS with UTF8)
    • support varies, but very good in aggregate
  • Tech admin: Analysis of logging, so any errors can be resolved promptly. If you are using a website as an active communication channel with your users; this is essential.

    • NB: This can be set as a HTML header or a HTTP header; depending on App details either may be better.

Less common security headers, that address specific issues

  • Add SNI ‚Äúserver name indicator‚Äù, or better ESNI ‚ÄúEncrypted SNI..." unless you have exclusive use of an IP pool 11 12. This reduces security warnings, as the browser knows who you are better. This should be already present. Support, SNI @ >98% 13, ESNI ??not found??
  • Attach X-Frame-Options to sensitive content to limit capacity for click-jacking attacks 14. This is deprecated by frame-ancestors in CSP, however still recommended usage for 'defence in depth'. According to MDN, this has no effect as a in-page META element.
  • Attach X-XSS-Protection to OBJECTS or FRAMES with external content where you want scripts to be inert 15. This is deprecated by CSP, however still recommended usage for 'defence in depth'.
  • X-Content-Type-Options: nosniff 16 to disable some browser features which can be dangerous.
  • Referrer-Policy strict-origin-when-cross-origin 17 will have most impact on social media-type sites, but forces reduced outbound data for the referrer header on later requests. Pay attention to spelling change.
  • Access-Control-Allow-Origin 18 the CORs setup header. See cors-headers for longer notes.
  • Cross-Origin-Opener-Policy (COOP) 19. I can't clearly explain this in one sentence, pls see link.
  • Cross-Origin-Embedder-Policy (COEP) 20
  • Cross-Origin-Resource-Policy (CORP) 21
  • Permissions-Policy example Permissions-Policy geolocation=(), camera=(), microphone=() 22 A header set by the webserver to disable JS features in the browser ~ for that browser tab only. MDN says not finalised.
  • X-Powered-By remove / overwrite this header, a waste of bytes, and exposing data that doesn't need to be shared. Unfortunately common.
  • X-AspNet-Version remove / overwrite this header, it's a waste of bytes, and exposing data that doesn't need to be shared.
  • X-AspNetMvc-Version leSigh microsoft, ditto
  • X-DNS-Prefetch-Control 23 should the standard browser feature DNS prefetch be enabled for this site?

Front-end performance improvements

I started this list as tidy-ups I have done on older established sites, and I just kept writing. I am defining performance as reducing content-reflows and render cost then reducing work measured in download bytes and time to display content. It is more difficult to write similar backend list, as there are many non-overlapping technologies available. These boring but reliable performance improvements should be present by-default in frameworks and app kits, but your millage may vary:

  • Use PICTURE 1 to select image to screen size and orientation, supported by 97.8% 2.
  • Check value supplied by any JS plugins, especially if you / your team controls local source.
  • Move fonts declarations to head of CSS, or maybe in HTML HEAD. Prune fonts formats to ones needed and supported. Favour local static hosting fonts, over dynamically generated ones, from another continent. A minor change to lettering appearance is much less urgent than fonts-for-iconsets. Fonts are a universe in their own right, see web-font-2024, utf8-richtext-maquette, fonts-ideal, colourise-emoji, symbol-tools. Supported browser ranges are listed in the attached articles.
  • Where this works with the design, favour UTF8 glyths for icons over an image. AND favour a SVG image over a JPEG / Bitmap or similar image. The UTF8 will be 1-4 bytes, and entirely inline but the image is often over a thousand bytes. The UTF8 will mostly match the colour scheme as a free step. I would suggest using default fonts for the UTF8 symbols, as not every font has every glyth. Look at colourise-emoji, for handling and limitations.
  • There are various bits of technology that were innovative and added value when they were written (most likely in JS), but now should be replaced with a browser feature. Remove the old ‚Äúcool tech‚Äù, as it now just junk dead-code eg, ie-shim.js should be removed with extreme prejudice, MSIE is removed by Microsoft on all machines with updates. Test carefully when making these removals. A properly loading JS Shim will assist the odd user that needs it, but more features are being moved into the w3c browser ‚Äúwidely supported‚Äù baseline standard.
    • Any full-screen image display widget in JS can be replaced by a few lines of CSS, and save download capacity, see modal-popover, zoomable-widget.
  • Try preloading essential iconset fonts / images for diagrams / CSS libs directly in the HTML HEAD with LINK rel=‚Äúpreload‚Äù 3 4 5 6. There are ‚Äúsolutions‚Äù describing making LINK elements with preload via JS. However, by the time the JS has been downloaded and parsed, it's not the early loading phase anymore ~ so I'm not sure how this helps. This tech is supported by 96.02% of browsers 7
  • Use loading=lazy 8 and height / width attributes on IMG elements, so the image(s) can be loaded last, but nothing will reflow. Currently supported by 96.31% of browsers 9
  • Reduce animation to the most import bit, rather than all over. Animation reduces readability, and the further we travel from 1995 the less people are impressed by it. For the important animation, platforms like MaterialUI are great for consistency and being able to deliver story points quickly, BUT maybe a little bit of CSS is smaller and faster e.g. 10. There are other technologies that could be used, but my point is, good DX often isn't performant.
  • Bundling and splitting CSS into ‚Äúall the CSS for this app/site and mobile and portrait‚Äù, then ‚Äúall the CSS for this app/site and tablet and landscape‚Äù, and ‚Äúall the CSS for this app/site and a big screen‚Äù may help reduce download volumes if the CSS is different in each case. This isn't true in all apps / sites. The screen orientation is the important metric, but the precise visualisations created should match your users, not some online notes. Assuming your CSS works in this architecture, media queries can be added to the STYLE element, so only relevant CSS is downloaded. There are many text centric sites where multiple files would be nearly identical so this step buys nothing. To reduce risk, there should be a default ‚Äúcatchall‚Äù style sheet with a flexible RWD / liquid layout. This needs to be a compile step, or the render testing will be very lengthy. Try PurgeCSS npm git, or PurityCSS npm git 11 12.
  • Look at the HTML after all the technology has happened. Try to migrate to use HTML DIALOG and DETAILS / SUMMARY with relevant CSS, as these are smoother, more secure, smaller, and faster than some 3rd party SPA Component. For modern CSS, many features in a HTML SELECT can be styled, so migrating to a real SELECT box is probably better UX.
  • Encourage marketing-centric plugins to load after the actual content. For certain vendors, I edit their JS snippet to use a DOM loadevent, then a small setTimeout, rather than HTML3 onload event. This means users get an interactive page load time that is 4-5s faster.
  • Check for usage of legacy features based on CSS margins, as this caused page reflows. Try CSS flex 13 or flex-grid 14. These should be a feature of your CSS framework. Assume global support for each CSS option mentioned in this paragraph. I don't have experience with CSS float making reflows, but that should be deprecated also.
  • Eliminate non-visible white-space from your HTML for production use on SSR or static sites. There are plugins for your framework, and/or a regex over the source code that makes the HTML. If you are using an SPA, this step is not important.
  • Ensure everything in minified correctly, and trees are shaken (i.e. check the file rather than looking at the file suffix). #leSigh Check fonts are compressed when they say they are. #leSigh Check images have correct resolution, correct JPEG compression and correct palette size, again #leSigh. In many cases, diagram-type images are smaller as SVG, but photo-type images need to be either WEBP or JPEG2000. I assert that there are no dodgy small resolution JPEGs for anything recent, so the newer format is better 15. Keep measuring everything.
  • I haven't mentioned deprecate table-based layouts from the early noughties with extreme prejudice... but do so if found.
